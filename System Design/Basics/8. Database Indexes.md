# Database Indexes - Relational vs NoSQL

## What is a Database Index?

An index is a **separate, sorted data structure** that helps the database find rows faster — without it, the database has to scan every single row (a **Full Table Scan**) to find matching data.

**Analogy:** A book's index page. Instead of flipping through all 1000 pages to find "Normalization," you check the index → "Page 245" → jump straight there.

An index stores a mapping of **column value → row location**, so lookups become much faster than scanning the whole table.

---

## Trade-offs of Using an Index

| Benefit | Cost |
|---|---|
| Much faster `SELECT` / read queries | Slower `INSERT` / `UPDATE` / `DELETE` — the index must be updated on every write |
| Faster `WHERE`, `JOIN`, `ORDER BY` operations | Extra disk space needed to store the index structure |

Because of this trade-off, indexes are **not added to every column** — only to columns that are frequently used in filtering, joining, or sorting.

---

## Relational Database Index vs NoSQL Index

| Aspect | Relational DB (MySQL, PostgreSQL) | NoSQL DB (MongoDB, Cassandra) |
|---|---|---|
| **Default structure** | Mostly **B-Tree** (sorted tree, great for range queries) | Also mostly **B-Tree** (e.g., MongoDB); some NoSQL databases use hash-based indexes |
| **Schema** | Fixed schema — table structure is predefined; indexes apply to specific columns | Flexible/schema-less — documents can have varying fields; index can be applied to any field, including nested fields |
| **Index type by data layout** | **Clustered Index**: actual data is physically stored in the order of the indexed column (usually the Primary Key, one per table). **Non-Clustered Index**: a separate structure that just points to the row location | Generally behave like non-clustered indexes; e.g., MongoDB automatically creates an index on `_id` |
| **Joins** | Indexes are critical because relational queries often involve **JOINs** across multiple tables, which are slow without proper indexing | Joins are less common (data is often denormalized/embedded within a single document), so indexing focuses more on single-query speed than join optimization |
| **Types of indexes** | Primary Key Index, Unique Index, Composite (multi-column) Index, Full-text Index | Single Field Index, Compound Index, **Multi-key Index** (can index values inside arrays — not possible in relational DBs), Geospatial Index, Text Index |
| **Scaling** | Indexing is optimized for a single server; becomes complex in distributed setups | Designed with distributed/sharded systems in mind — indexing works alongside sharding across multiple nodes |

---

## Key Takeaway

> In a **Relational DB**, indexes mostly speed up lookups on fixed rows/columns since the schema is rigid.
> In **NoSQL**, indexes can be applied flexibly — even inside nested documents or arrays (e.g., MongoDB's **Multi-key Index** can index individual items inside an array), which isn't possible in a traditional relational structure.
