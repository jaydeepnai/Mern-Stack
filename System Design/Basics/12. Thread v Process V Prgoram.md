# Program vs Process vs Thread — Notes

## Definitions

| Term | Definition |
|---|---|
| **Program** | A **passive** entity — a set of instructions stored on disk (e.g., an executable file). Does nothing by itself. |
| **Process** | An **active instance** of a program in execution — has its own memory, registers, and resources managed by the OS. |
| **Thread** | The **smallest unit of execution** within a process — shares memory/files with other threads of the same process, but has its own stack and program counter. |

---

## Characteristics

### Program
- Static and passive — stored on disk in binary/executable form.
- Consumes **no CPU or memory** until it's loaded and actually executed.
- **One program** can spawn **multiple processes** (e.g., opening the same app twice → two processes, one program).

### Process
- Dynamic and active — has its own memory space, stack, and registers.
- **Isolated** from other processes → better security/reliability, but **higher overhead**.
- Switching between processes (**context switching**) is expensive because the OS must save/restore the entire process state.

### Thread
- Lightweight, exists **within** a process.
- Shares the same code, data, and file resources as sibling threads — but has its **own stack and program counter**.
- Thread creation/switching is **much faster** than process switching.
- Ideal for: real-time updates, parallel computations, handling multiple client requests concurrently (e.g., a web server handling many connections).

---

## Comparison Table

| Aspect | Program | Process | Thread |
|---|---|---|---|
| **Definition** | Passive set of instructions | Active execution of a program | Smallest execution unit within a process |
| **Nature** | Static, passive | Dynamic, active | Lightweight, active |
| **Execution** | Not executed directly | Executed by CPU | Executed within a process's context |
| **Memory** | No memory allocated | Own dedicated memory space | Shares memory with parent process |
| **Isolation** | Independent | Isolated from other processes | Not isolated — can affect sibling threads |
| **Lifespan** | Exists permanently until deleted | Lives until execution finishes | Lives until its task within the process is done |
| **Communication** | N/A | Needs Inter-Process Communication (IPC) | Direct communication via shared memory |
| **Overhead** | None | High (costly context switching) | Low (fast context switching) |

## Why This Matters
- **Program** defines *what* should be executed.
- **Process** ensures execution happens in an **isolated, resource-managed** environment.
- **Thread** enables **concurrency/parallelism** efficiently within that process.

This layered structure (Program → Process → Thread) is how modern operating systems achieve **multitasking, parallelism, and responsiveness**.

---

## Quick Recap
> **Program** = code sitting on disk, doing nothing.
> **Process** = that program actually running, with its own isolated memory (expensive to switch).
> **Thread** = a lightweight worker inside a process, sharing memory with siblings (cheap to switch, great for concurrency).
