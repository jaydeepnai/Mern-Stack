# Relational Model in DBMS

The **Relational Model** is a way of organizing and storing data using **tables** (also called *relations*). It was introduced by **Edgar F. Codd in 1970** and is the foundation of almost all modern databases (MySQL, Oracle, PostgreSQL, etc.).

Think of it like an **Excel sheet** — data is stored in rows and columns instead of files or free-form text.

---

## Key Terminologies

### 1. Table (Relation)
A table represents **one specific entity or concept** — like a single Excel sheet. Example: a table named `STUDENT` holds only student-related data.

### 2. Row (Tuple)
A single **horizontal record** in the table. Each row represents **one instance** of the entity — e.g., one student's full details.

### 3. Column (Attribute)
A single **vertical field** in the table. Each column represents **one property** of the entity, and all values in that column must be of the same data type (e.g., `AGE` always holds numbers).

### 4. Degree
The **number of columns (attributes)** in a table.
> Example: `STUDENT` has 5 columns (`ROLL_NO`, `NAME`, `ADDRESS`, `PHONE`, `AGE`) → Degree = **5**

### 5. Cardinality
The **number of rows (tuples)** in a table.
> Example: 4 student records → Cardinality = **4**

## Example: STUDENT Table

| ROLL_NO | NAME | ADDRESS | PHONE | AGE |
|---|---|---|---|---|
| 1 | Akshay | Gurgaon | 6487489939 | 18 |
| 2 | Yuvaan | Delhi | 7438489027 | 20 |
| 3 | Kuhuk | Jaipur | 3763287819 | 20 |
| 4 | Gauri | Pune | 2397328659 | 19 |

- Columns (`ROLL_NO`, `NAME`, `ADDRESS`, `PHONE`, `AGE`) = **Attributes**
- Each row = **one student's record (Tuple)**

---

## Data Integrity

Integrity rules are checked **before any operation** (insert, delete, update). If a rule is violated, the operation fails.

| Type | Meaning |
|---|---|
| **Domain Integrity** | Every value in a column must match the correct data type (e.g., `AGE` must always be a number, never text). |
| **Entity Integrity** | Every row must be **unique** — enforced using keys (Primary Key, Candidate Key, Super Key, etc.). No two rows can be identical/duplicate. |
| **Referential Integrity** | If a table references another table (via a **Foreign Key**), that reference must always point to a **valid, existing** row in the other table. |

---

## Atomic Values

Every value stored in a column must be **atomic** — meaning it cannot be broken down further.

**Bad:**
```
NAME
Akshay Kumar Sharma
```

**Good (Atomic):**
```
FIRST_NAME | MIDDLE_NAME | LAST_NAME
Akshay     | Kumar       | Sharma
```

Splitting composite values into atomic columns makes data easier to search, filter, and manage.

---

## Quick Recap
- **Table** = Entity | **Row** = one record | **Column** = one property
- **Degree** = # of columns | **Cardinality** = # of rows
- **Integrity types**: Domain (correct data type) → Entity (unique rows via keys) → Referential (valid foreign key references)
- Values must be **atomic** (not further divisible)
