
# NoSQL Databases & Cassandra — Notes

## SQL vs NoSQL — How Data is Stored

**SQL:** A `Person` table stores simple fields, but complex data (like `address`) is split into a **separate table**, linked via a **Foreign Key** (`address_id`). Retrieving full info requires a **JOIN**.

**NoSQL:** All related data — name, age, role, and even the address — is stored together as **one JSON document (blob)**, with `address` nested directly inside. No separate table, no foreign key.

> **Analogy:** SQL is like a filing cabinet with separate drawers (you open multiple drawers via a join). NoSQL is like one complete file where everything is already stapled together.

---

## Advantages of NoSQL

### 1. Fast Inserts & Retrievals
User data (name, age, address) is usually inserted together and retrieved together (`SELECT *`). Since NoSQL keeps it all in one blob, there's no need for joins or scattered column reads — insert once, retrieve once.

### 2. Flexible Schema
- In SQL, adding a new column (e.g., `salary`) requires **locking the entire table** — expensive and risky for consistency.
- In NoSQL, you can simply start adding the new field to new documents. Older documents remain unaffected since there's no enforced schema.

### 3. Built-in Horizontal Partitioning (Sharding)
NoSQL databases are designed for scale — data is automatically distributed across multiple nodes/machines, prioritizing **availability**.

### 4. Optimized for Aggregations
Well-suited for computing metrics like average age, total sales, etc., across large datasets.

---

## Disadvantages of NoSQL

| Problem | Explanation |
|---|---|
| **Consistency issues** | Different nodes may temporarily hold different data for the same ID — ACID guarantees are not provided. This is why financial/transactional systems avoid NoSQL. |
| **Not read-optimized** | To fetch a single field (e.g., age), the database must read the **entire blob** first, then extract the field — slower than reading a single SQL column directly. |
| **No implicit relations** | Unlike SQL (where "R" = Relational, enforced via foreign keys), NoSQL has no built-in way to enforce relationships between records. |
| **Joins are hard** | Joining two NoSQL collections requires manually scanning through data blocks to match values — there's no built-in join engine like in SQL. |

## Cassandra — Sharding & Load Distribution

Cassandra is a popular NoSQL database, useful for understanding these concepts practically.

**Setup:** A cluster of nodes (e.g., 5 nodes). Every request/key is passed through a **hash function**, which returns a value used to determine which node stores that data.

- **Good hash function** → requests are distributed **uniformly** across all nodes → balanced load.
- **Bad hash function** → requests cluster around a few values → one node gets overloaded and may crash.

**Fix — Multi-level Sharding:** If one node becomes a bottleneck, route its traffic into a **second, smaller cluster** with its own separate hash function to further distribute the load.

---

## Replication & Quorum

### Replication
When a request lands on a node (say Node 2), the data is also copied to the **next node(s)** in the ring (based on the replication factor). This protects against data loss if a node crashes.

**Benefits:**
- Faster reads (any replica can respond)
- Safer writes (data survives even if one node fails)

### Quorum
When reading data, Cassandra doesn't just ask one node — it asks a set of replicas and relies on the **majority (quorum)** to decide the correct answer.

> **Analogy:** Like asking 3 witnesses about an event — if 2 agree, that's taken as the truth, even if the 3rd is unavailable.

- If quorum = 2 (out of 3 replicas) and 2 agree → the value is accepted and returned.
- If there's no majority agreement (e.g., replicas disagree or are unreachable) → Cassandra returns a **database error** (not a false "not found") so the application knows something is wrong.

This design favors **availability over strict consistency** — a small, rare risk that NoSQL systems consciously accept.

---

## How Cassandra Writes & Stores Data Internally

1. **In-memory Log Write** — New writes are appended sequentially to an in-memory log — very fast since there's no searching involved.
2. **Flush to SSTable** — Periodically, this in-memory data is flushed to disk as an **SSTable (Sorted String Table)** — keys are stored in sorted order. SSTables are **immutable** (never edited after creation).
3. **Multiple SSTables for Same Key** — If the same key is updated later, a new SSTable entry is created rather than modifying the old one. The **timestamp** determines which version is the latest/correct one.
4. **Compaction** — Periodically, multiple SSTables are merged (similar to a merge sort) to eliminate duplicate/stale entries and reclaim storage space.
5. **Tombstones (Deletes)** — Deleting a record doesn't remove data immediately; instead, a **tombstone flag** marks it as dead. Any read that encounters a tombstone treats the record as deleted.

---

## Quick Recap

- **NoSQL:** All data in one flexible document → fast inserts/reads of full records → schema-less → built for horizontal scale → weaker on consistency, reads of single fields, and joins.
- **Cassandra:** Hash-based sharding distributes load → replication ensures data safety → quorum decides the correct answer during reads → writes go to an in-memory log → flushed to immutable SSTables → cleaned up via compaction and tombstones.
