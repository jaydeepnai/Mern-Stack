# Thrashing (Operating Systems) — Notes

## Definition
**Thrashing** occurs when a process spends **more time paging** (swapping pages in/out of memory) **than actually executing instructions**. It shows up as a **sharp drop in CPU utilization**, usually caused by too high a degree of multiprogramming.

---

## How It Happens (The Vicious Cycle)

1. More processes are added to the system to **increase CPU utilization**.
2. But available memory frames are limited — processes now need **more frames than are available**.
3. This causes **frequent page faults** (a process needs a page that isn't currently in memory).
4. Frequent page faults create a **long queue for the paging device**, slowing everything down further.
5. CPU utilization drops — the scheduler (misreading this as "CPU is idle, add more work") **adds even more processes**.
6. This makes memory pressure worse → even more page faults → CPU utilization drops further.

This creates a self-reinforcing loop of constant paging with almost no real execution — this loop **is** thrashing.

---

## Causes of Thrashing

- **High degree of multiprogramming** exceeding available memory frames.
- Processes repeatedly requesting pages **not currently in memory** → frequent page faults.
- Excessive paging **eats into CPU execution time**, reducing overall system throughput.
- New processes **steal memory frames** from existing running processes, triggering even more page faults for those processes.

---

## Working Set Model (Solution)

Based on the principle of **locality of reference** — a process tends to actively use only a small subset of its pages at any given time.

- **Working Set:** The set of pages a process is actively using within a time window **Δ (delta)**.
- If a page was used within the last Δ time units, it belongs to the process's working set.
- **WSS (Working Set Size):** Number of frames a process needs to avoid faulting — essentially the size of its working set.
- **Total Demand for Frames:**
  ```
  D = Σ WSSi   (sum of working set sizes of all running processes)
  ```
- **Thrashing occurs when D > total available memory frames** — i.e., total demand exceeds what the system can supply.

---

## How to Minimize Thrashing

| Strategy | How it Helps |
|---|---|
| **Local Replacement** | When a process needs a new page, it can only replace *its own* pages — not steal frames from other processes. This prevents cascading page faults across the system. |
| **Working Set Strategy** | Ensures each process is allocated enough frames to hold its active working set, keeping CPU utilization high while avoiding excessive paging. |

---

## Quick Recap
- **Thrashing** = more time spent paging than executing → CPU utilization crashes.
- **Root cause:** Too many processes competing for too few memory frames → frequent page faults → scheduler adds more processes → worsens the problem.
- **Working Set Model:** Track each process's actively-used pages (within window Δ); if total demand for frames (`D = Σ WSSi`) exceeds available memory, thrashing occurs.
- **Prevention:** Local replacement policy + working set-based frame allocation.
