# Consistent Hashing — Why Simple Hashing Fails (Part 1: The Problem)

## The Basic Setup

Imagine you built an algorithm (e.g., facial recognition) running on one server. It gets so popular that one server can't handle all the requests, so you add more servers. Now the question becomes: **which server should handle which request?**

This is the classic **Load Balancing** problem — distribute incoming requests evenly across all servers so no single server is overloaded.

---

## The Simple (Naive) Solution: Hash + Modulo

Each request has a **Request ID**. The basic approach:

1. Take the Request ID and pass it through a **hash function** → get a number.
2. Take that number **modulo N** (where N = number of servers).
3. Send the request to the server matching that result.

**Example with 4 servers (S0, S1, S2, S3):**
- `hash(R1) = 3` → `3 % 4 = 3` → goes to Server 3
- `hash(R2) = 15` → `15 % 4 = 3` → goes to Server 3
- `hash(R3) = 12` → `12 % 4 = 0` → goes to Server 0

Since the hash function is uniformly random, the load naturally balances out — each server gets roughly `1/N` of the total requests. Looks perfect... until you need to scale.

---

## The Problem: Adding a New Server Breaks Everything

Suppose your app goes viral and you need to add a **5th server**. Now every calculation changes because you're now doing **modulo 5** instead of **modulo 4**:

- `hash(R1) = 3` → `3 % 5 = 3` (same server — lucky!)
- `hash(R2) = 15` → `15 % 5 = 0` (changed! was server 3, now server 0)
- `hash(R3) = 12` → `12 % 5 = 2` (changed! was server 0, now server 2)

**Visualizing it as a pie chart:** With 4 servers, each one "owns" 25% of the possible hash values. When a 5th server is added, each server should now own only 20%. But the naive `mod N` approach doesn't just shift a small slice to the new server — it **reshuffles almost everything**.

In the video's worked example, adding just **one** server to a 4-server system caused close to **100% of all requests to be remapped** to different servers — essentially the entire system changed.

---

## Why This Is a Big Deal (Not Just a Math Inconvenience)

In real systems, Request IDs aren't random — they're often based on something like **User ID**. This means:
- The same user always hashes to the same value → always routed to the same server.
- This lets you **cache** that user's data locally on that server (e.g., their profile), avoiding repeated database lookups — a smart optimization.

**But when adding a server causes a massive remap:**
- Nearly all users get sent to a *different* server than before.
- All the cached data built up for those users becomes **useless** overnight.
- This causes a huge spike in cache misses and database load right when you're trying to scale — the opposite of what you want.

---

## What We Actually Want Instead

When scaling from 4 servers to 5, the *ideal* behavior is:
- Take a **small slice** from each existing server (just enough to give the new server its fair 20% share).
- **Minimize the total number of requests that get remapped** — most requests should keep going to the same server as before.

The naive `hash % N` approach cannot guarantee this — a small change in `N` can cause a massive, unpredictable change in server assignment.

**This is exactly the problem Consistent Hashing is designed to solve** — it allows scaling the number of servers up or down while remapping only a small, minimal fraction of requests, keeping caches mostly intact.

> Note: This part explains *why* consistent hashing is needed. The actual mechanism (hash ring, virtual nodes) is typically covered as the next step/topic.

---

## Quick Recap
- **Goal:** Evenly distribute requests across servers (load balancing).
- **Naive approach:** `hash(request_id) % N` → simple, but breaks badly when N changes.
- **Real problem:** Adding/removing a server can remap almost all requests, destroying caches built around request/user IDs.
- **What's needed:** A hashing strategy where scaling up/down causes only a **minimal, proportional** remapping — this is what Consistent Hashing provides.
